## 1. First Principles: Theory vs Practice

### 1.1 The Fundamental Asymmetry

```
THEORY (Knowledge Graph)          PRACTICE (Reflection Graph)
─────────────────────────         ─────────────────────────
What SHOULD happen                What ACTUALLY happened
Decisions, patterns, rules        Successes, failures, insights
Authored by humans                Generated by execution
Versioned by intent               Versioned by evidence
Static until manually updated     Continuously accumulating
Confidence from reasoning         Confidence from repetition
```

The current framework has only the left column. The Reflection Engine adds the right column. Both must exist. Neither is sufficient alone.

### 1.2 What This System Is NOT

| Not This | Because |
|---|---|
| **Chat memory** | Conversations are ephemeral and noisy. Reflections are structured observations. |
| **Conversation history** | History is serial and chronological. Reflections are relational and queryable. |
| **RL/RLHF** | We don't train models. We evolve knowledge structures. |
| **Auto-ML** | No automatic model selection or hyperparameter tuning. |
| **Fine-tuning pipeline** | This system evolves knowledge, not model weights. |
| **Log aggregation** | Logs are raw. Reflections are semantic, analyzed, and structured. |

### 1.3 What This System IS

The Reflection Engine is a **deterministic observation system** that:

1. **Watches** every completed engineering task
2. **Compares** what happened against what was expected
3. **Records** structured observations (reflections)
4. **Accumulates** evidence over time (hundreds to thousands of reflections)
5. **Detects** recurring patterns in the accumulated evidence
6. **Generates** validated hypotheses (improvement candidates)
7. **Proposes** specific changes to knowledge, compiler rules, or policies
8. **Requires** governance approval before any change takes effect

### 1.4 Design Principles

| Principle | Meaning |
|---|---|
| **Evidence over intuition** | One failure is noise. Fifty failures of the same pattern is a signal. The system accumulates, never reacts. |
| **Observation over modification** | The engine observes and recommends. It never directly modifies knowledge, compiler rules, or policies. |
| **Structured over prose** | Every reflection is machine-queryable. No free-text summaries. No "the model struggled with..." narratives. |
| **Accumulation over replacement** | Every reflection is preserved forever. Knowledge evolves by adding evidence, never by rewriting history. |
| **Traceability over convenience** | Every knowledge change is traceable back to the specific reflections that motivated it. Every reflection traces back to the task that produced it. |
| **Safety over speed** | A bad knowledge update costs more than a thousand missed optimizations. Gates are mandatory. |
| **Determinism over probability** | The reflection engine itself is deterministic — same input → same reflection. Synthesis can use statistics, but every decision is explainable. |
| **Separability over coupling** | The Reflection Engine is a distinct component. It can be disabled with zero impact on the rest of the framework. |

## 2. The Reflection Engine

### 2.1 Position in the Framework

```
┌──────────────────────────────────────────────────────────┐
│                  ENGINEERING EXECUTION                    │
│                                                           │
│  User Request → Orchestrator → Compiler → Subagent →     │
│  Implementation → Review → Test → Complete                │
│                                                           │
└──────────────────────────┬───────────────────────────────┘
                           │ execution complete
                           ▼
┌──────────────────────────────────────────────────────────┐
│                 REFLECTION ENGINE (triggered)              │
│                                                           │
│  1. Collect execution metadata                            │
│  2. Compare expectations vs. reality                      │
│  3. Generate structured reflections                       │
│  4. Store in reflection graph                             │
│  5. Update metrics                                        │
│                                                           │
└──────────────────────────┬───────────────────────────────┘
                           │ reflections stored
                           ▼
┌──────────────────────────────────────────────────────────┐
│              SYNTHESIS ENGINE (periodic)                   │
│                                                           │
│  1. Query reflection graph                                │
│  2. Cluster similar observations                          │
│  3. Detect recurring patterns                             │
│  4. Generate hypotheses                                   │
│  5. Produce improvement candidates                        │
│                                                           │
└──────────────────────────┬───────────────────────────────┘
                           │ candidates proposed
                           ▼
┌──────────────────────────────────────────────────────────┐
│              EVOLUTION PIPELINE (gated)                    │
│                                                           │
│  Candidate → Validate → Experiment → Approve → Promote    │
│                                                           │
└──────────────────────────────────────────────────────────┘
```

### 2.2 Trigger Points

The Reflection Engine fires automatically after:

| Trigger | Description | Frequency |
|---|---|---|
| **Task complete** | Builder finishes a task, reviewer approves | Every task |
| **Contract complete** | All tasks in a contract are done | Every contract |
| **Review failure** | Reviewer rejects Builder output | On rejection |
| **Build failure** | CI fails (lint, typecheck, test) | On failure |
| **Compiler invocation** | Compiler is called (records compilation quality) | Every compile |
| **Session end** | Session compaction or closure | Every session |
| **Knowledge atom query** | Atom was loaded but unused by the model | On detection |

### 2.3 What Gets Reflected

The engine collects structured input from multiple sources:

```
EXECUTION INPUT:
  ├── Task description (original user request)
  ├── Execution contract (if applicable)
  │   ├── Expected capabilities
  │   ├── Expected knowledge modules
  │   ├── Expected files changed
  │   └── Expected patterns applied
  ├── Compiled context
  │   ├── Resolved capabilities
  │   ├── Selected atoms (which were loaded)
  │   ├── Token budget
  │   ├── Compression applied
  │   └── Compiler metadata
  ├── Agent type and model
  └── Project state at start

EXECUTION PROCESS:
  ├── Tool calls made
  │   ├── read calls (files accessed)
  │   ├── edit/write calls (files changed)
  │   ├── bash calls (commands executed)
  │   ├── skill calls (atoms actually loaded vs. recommended)
  │   └── task calls (subagents dispatched)
  ├── Knowledge atoms loaded vs. used vs. unused
  ├── Compiler passes executed
  └── Subagent invocations

EXECUTION OUTPUT:
  ├── Files changed (actual diff)
  ├── Test results (pass/fail/counts)
  ├── Review outcome (passed/failed/revisions required)
  ├── Build status (lint, typecheck, build)
  ├── Commit created
  └── State files updated

EXECUTION QUALITY:
  ├── Contract compliance (% of acceptance criteria met)
  ├── Unexpected errors (errors not caught by knowledge)
  ├── Clarification requests (did the model ask for help?)
  ├── Review iterations (how many review rounds?)
  ├── Token efficiency (actual vs. budget)
  └── Time to complete
```

## 3. Reflection Model & Schema

### 3.1 Reflection Atom

A reflection is a structured observation — a machine-queryable data point, not a prose summary. It follows the same pattern as knowledge atoms but records experience rather than standards.

```yaml
---
# ── IDENTITY ──────────────────────────────────────────────
id: string                    # REFL-{timestamp}-{hash8}
                              # Example: REFL-20260712-a3f2b1c4
type: reflection

# ── CLASSIFICATION ────────────────────────────────────────
reflection_type:              # What kind of observation is this?
  success | failure | insight | inefficiency |
  gap | pattern_confirmed | pattern_refuted |
  knowledge_sufficient | knowledge_insufficient |
  compiler_efficient | compiler_wasteful |
  policy_helpful | policy_harmful

# ── ORIGIN ────────────────────────────────────────────────
task_id: string               # Contract ID or task identifier
task_type: string             # feature, bug, refactor, etc.
session_id: string            # OpenCode session ID
agent_type: string            # orchestrator, builder, etc.
model_id: string              # Which model was used
generated_at: ISO8601         # When this reflection was created
generated_by: string          # "reflection-engine:v1.0.0"

# ── OBSERVATION ───────────────────────────────────────────
# The actual structured observation. One per reflection.
# Never prose. Always this format:

category:                     # What domain does this concern?
  knowledge | compiler | policy | process | testing | review

subject:                      # Specific thing observed
  type: atom_id | pass_name | policy_name | capability | pattern | tool
  id: string                  # The specific ID (atom ID, pass name, etc.)

observation:                  # What was observed
  expected: string            # What should have happened (from knowledge/contract)
  actual: string              # What actually happened (from execution)
  delta: string               # The gap between expected and actual

# ── EVIDENCE ──────────────────────────────────────────────
evidence:
  execution_id: string        # Unique execution identifier
  tool_call_count: number     # Total tool calls made
  tokens_used: number         # Actual tokens consumed
  tokens_budget: number       # Budget allocated
  files_changed: string[]     # Files modified
  test_results:
    passed: number
    failed: number
  review_iterations: number   # Rounds of review needed
  compilation_metadata:       # From the compiler
    passes: object
    atoms_loaded: string[]    # Which atoms were provided
    atoms_used: string[]      # Which atoms were actually referenced
    atoms_unused: string[]    # Which atoms were loaded but never used
    atoms_missing: string[]   # Which capabilities had no atoms

# ── SIGNAL ────────────────────────────────────────────────
confidence: high | medium | low
                              # How confident is this observation?
                              # high = clear causal link
                              # medium = strong correlation
                              # low = possible coincidence

severity: critical | major | minor | info
                              # How important is this observation?

frequency: number             # How many times has this category been observed?
                              # (computed by synthesis, not set at creation)

# ── RELATIONSHIPS ─────────────────────────────────────────
# Connects this reflection to the rest of the framework
relationships:
  knowledge_atoms: string[]    # Atom IDs this observation concerns
  compiler_passes: string[]    # Compiler passes involved
  policies: string[]           # Policies affected
  capabilities: string[]       # Capabilities involved
  decisions: string[]          # Decision atoms relevant
  patterns: string[]           # Pattern atoms relevant
  rules: string[]              # Rule atoms relevant

  related_reflections: string[]  # Other reflections on same subject
  superseded_by: string | null   # If this reflection's conclusion was updated

# ── CONTEXT ───────────────────────────────────────────────
# Optional additional context
contract_reference: string | null   # Link to contract.md section
error_message: string | null        # If this reflects a failure
environment:
  framework_version: string
  compiler_version: string
  knowledge_graph_version: string
---

# {Human-readable title summarizing the observation}

## Expected

{What the framework expected to happen, based on knowledge atoms, contract, policies}

## Actual

{What actually happened during execution, with specific data points}

## Delta

{The difference — what can be learned}
```

  policies: []
  capabilities:

    - DEC-AUTH-001

    - PAT-jwt-rotation

## 4. Reflection Types

### 4.1 Complete Taxonomy

| Type | Trigger | Example |
|---|---|---|
| `success` | Task passes all checks on first attempt | "Pattern PAT-jwt-rotation applied correctly. Contract fully satisfied in one pass." |
| `failure` | Task fails review, tests, or build | "Reviewer rejected implementation. Missing atomic transaction for token invalidation." |
| `insight` | Unexpected positive or negative pattern | "Compiler pass 3 resolved dependencies that were already covered by pass 2 — redundant." |
| `inefficiency` | Token waste, unnecessary tool calls, excessive iterations | "PAT-repository loaded (95 tokens) but never referenced. Not needed for auth-only tasks." |
| `gap` | Missing knowledge atom or capability | "No atom exists for WebSocket authentication. Model improvised." |
| `pattern_confirmed` | Pattern applied N times without issues | "PAT-pagination has been applied 47 times with zero revisions. Confidence: established." |
| `pattern_refuted` | Pattern consistently fails in practice | "PAT-circuit-breaker caused 12 deadlocks in the last month. May conflict with PAT-repository locking." |
| `knowledge_sufficient` | All loaded atoms were used; no gaps | "Compiled context fully covered the task. 5 atoms loaded, 5 atoms used." |
| `knowledge_insufficient` | Model needed knowledge not in compiled context | "Builder asked 'how do I handle rate limiting?' — DEC-RATE-LIMIT-001 not loaded." |
| `compiler_efficient` | Compiler produced optimal context | "3 atoms loaded, all used, 40% under budget. Cache hit on stable prefix." |
| `compiler_wasteful` | Compiler loaded unnecessary atoms | "PAT-deployment loaded for a backend-only task. Capability 'deployment' incorrectly inferred from keyword 'env'." |
| `policy_helpful` | Policy prevented a bad outcome | "policy-enforcer blocked read of .env file. Prevented potential secret leak." |
| `policy_harmful` | Policy caused unnecessary friction | "Policy blocked 'git push' for a feature branch. Required manual override 3 times this session." |

### 4.2 Reflection Priority Matrix

| Type | Default Severity | Requires Synthesis? | Generates Candidate? |
|---|---|---|---|
| `failure` | major | Yes (N ≥ 3) | Yes (if recurring) |
| `gap` | critical | Yes (N ≥ 1) | Yes (immediately) |
| `pattern_refuted` | critical | Yes (N ≥ 5) | Yes (urgent) |
| `inefficiency` | minor | Yes (N ≥ 10) | Yes (if total savings > 5000 tokens) |
| `insight` | minor | Yes (N ≥ 5) | Maybe |
| `success` | info | No | No (confirms existing knowledge) |
| `pattern_confirmed` | info | Yes (N thresholds) | No (reinforces, doesn't change) |
| `knowledge_sufficient` | info | No | No |
| `compiler_efficient` | info | No | No |
| `policy_harmful` | major | Yes (N ≥ 3) | Yes |

## 5. The Reflection Graph

### 5.1 Why a Graph

Reflections are not isolated. They connect to everything in the framework:

```
                    ┌──────────────────────────┐
                    │    KNOWLEDGE GRAPH         │
                    │  (what should happen)      │
                    │                            │
                    │  DEC-* PAT-* RUL-* CAP-*  │
                    └──────────┬───────────────┘
                               │
                    ┌──────────▼───────────────┐
                    │    REFLECTION GRAPH        │
                    │  (what actually happened)  │
                    │                            │
                    │  REFL-* (observations)     │
                    └──────────┬───────────────┘
                               │
            ┌──────────────────┼──────────────────┐
            │                  │                  │
            ▼                  ▼                  ▼
   ┌────────────────┐ ┌──────────────┐ ┌────────────────┐
   │ COMPILER GRAPH │ │POLICY GRAPH  │ │ EXECUTION GRAPH│
   │ (passes, rules)│ │(policies)    │ │ (tasks, contract│
   └────────────────┘ └──────────────┘ └────────────────┘
```

### 5.2 Relationship Types

```
REFLECTION → KNOWLEDGE:
  concerns_atom          "This reflection is about PAT-jwt-rotation"
  confirms_atom          "This reflection confirms DEC-AUTH-001 is correct"
  contradicts_atom       "This reflection suggests PAT-pagination is incomplete"
  reveals_gap_for        "This reflection shows no atom exists for this capability"

REFLECTION → COMPILER:
  concerns_pass          "This reflection is about pass 3 (dependency resolution)"
  reveals_inefficiency   "Pass 5 (dedup) removed 0 atoms — redundant for small tasks"
  suggests_tuning        "Token budget for builder should be 3500, not 3000"

REFLECTION → REFLECTION:
  related_to             "Another reflection on the same subject"
  supports               "This reflection provides additional evidence for that one"
  contradicts            "These reflections disagree — investigate"
  superseded_by          "This reflection's conclusion was updated by a later one"
  clusters_with          "Belongs to the same synthesis cluster"

REFLECTION → TASK:
  generated_from         "This reflection came from CONTRACT-001"
  caused_by_task_type    "Bug tasks produce more 'failure' reflections than features"

REFLECTION → POLICY:
  concerns_policy        "This reflection is about policy-enforcer"
  blocked_by_policy      "Policy blocked an action during this task"
  suggests_policy_change "Policy X should be relaxed for agent type Y"
```

### 5.3 Graph Traversal Examples

```
Query: "Show me all reflections about PAT-jwt-rotation"
  → Traverse: REFL[concerns_atom=PAT-jwt-rotation]
  → Result: 47 reflections (12 failures, 30 successes, 5 inefficiencies)

Query: "What patterns produce the most failures?"
  → Traverse: REFL[type=failure] → concerns_atom → group by atom_id
  → Result: PAT-circuit-breaker (12), PAT-migration (8), PAT-jwt-rotation (3)

Query: "What compiler passes have never been flagged for inefficiency?"
  → Traverse: REFL[type=inefficiency, concerns_pass] → group by pass_name
  → Result: Pass 1 (parse), Pass 8 (validate), Pass 9 (output) — passes 2-7 all flagged

Query: "What knowledge gaps have been discovered but never filled?"
  → Traverse: REFL[type=gap] → reveals_gap_for → filter where no atom exists
  → Result: 14 gaps discovered, 9 filled, 5 still open
```

## 6. Reflection Lifecycle

### 6.1 States

```
┌──────────┐
│ created  │  Generated by the engine after task completion
└────┬─────┘
     │
     ▼
┌──────────┐
│  raw     │  Initial state. Not yet validated.
└────┬─────┘
     │ automated validation passes
     ▼
┌──────────┐
│validated │  Schema valid, no contradictions with known facts.
└────┬─────┘
     │ synthesis engine processes it
     ▼
┌──────────┐
│clustered │  Grouped with similar reflections by synthesis.
└────┬─────┘
     │ cluster reaches significance threshold
     ▼
┌──────────┐
│ signaled │  Part of a significant pattern. Eligible for candidate generation.
└────┬─────┘
     │ candidate created from this signal
     ▼
┌──────────┐
│ promoted │  Contributed to an improvement candidate that was promoted.
└────┬─────┘
     │ (any point — if superseded by a newer reflection)
     ▼
┌──────────┐
│superseded│  A newer reflection on the same subject provides better evidence.
└──────────┘
```

### 6.2 Lifecycle Rules

1. **`created` → `raw`**: Immediate. Engine outputs raw reflections.
2. **`raw` → `validated`**: Automatic. Schema validation passes.
3. **`validated` → `clustered`**: Periodic. Synthesis engine runs, clusters similar reflections.
4. **`clustered` → `signaled`**: Threshold-based. Cluster size ≥ significance threshold.
5. **`signaled` → `promoted`**: Gated. Candidate was approved and knowledge was updated.
6. **`*` → `superseded`**: Automatic. A newer reflection with higher confidence on the same subject.

## 7. Reflection Validation

### 7.1 Validation Checks

```
function validateReflection(reflection: Reflection): ValidationResult {
  let checks = []

  // Schema validation
  if (!schema.validate(reflection)) {
    return fail("Schema validation failed")
  }

  // Referential integrity: do referenced atoms exist?
  for (let atomId of reflection.relationships.knowledge_atoms) {
    if (!registry.hasAtom(atomId)) {
      checks.push(warn(`Atom ${atomId} not found in registry`))
    }
  }

  // Referential integrity: does the task exist?
  if (reflection.task_id && !state.hasContract(reflection.task_id)) {
    checks.push(warn(`Task ${reflection.task_id} not found`))
  }

  // Confidence vs evidence: high confidence requires strong evidence
  if (reflection.confidence === 'high' && !hasStrongEvidence(reflection)) {
    checks.push(warn("High confidence claimed without sufficient evidence"))
  }

  // No self-contradiction
  if (reflection.reflection_type === 'knowledge_sufficient' &&
      reflection.evidence.compilation_metadata.atoms_missing.length > 0) {
    checks.push(error("Claims 'knowledge_sufficient' but atoms_missing is non-empty"))
  }

  // Severity appropriateness
  if (reflection.reflection_type === 'success' && reflection.severity === 'critical') {
    checks.push(warn("Success reflection should not be critical severity"))
  }

  return { passed: checks.every(c => c.level !== 'error'), checks }
}
```

## 8. Reflection Confidence

### 8.1 Confidence Levels

| Level | Definition | Required Evidence |
|---|---|---|
| **high** | Clear causal link between observation and outcome | Same pattern observed in 5+ executions. Control: counter-examples exist (pattern absence → better outcome). |
| **medium** | Strong correlation, plausible causation | Same pattern observed in 3+ executions. Alternative explanations not ruled out. |
| **low** | Possible correlation, single observation | Single execution. May be coincidence. Worth tracking but not actionable alone. |

### 8.2 Confidence Evolution

A reflection's confidence can increase over time as supporting evidence accumulates:

```
REFL-001 (initial): confidence=low, frequency=1
  → 3 more similar reflections appear
  → Synthesis engine: "pattern detected with 4 observations"
  → REFL-001 updated: confidence=medium, frequency=4
  → 6 more similar reflections appear
  → Synthesis engine: "pattern confirmed with 10 observations over 3 projects"
  → REFL-001 updated: confidence=high, frequency=10
  → Candidate generated
```

### 8.3 Confidence-Computed Fields

```typescript
interface ConfidenceMetrics {
  observations: number          // How many reflections report this pattern?
  unique_projects: number       // Across how many different projects?
  unique_models: number         // Across how many different models?
  unique_agents: number         // Across how many different agent types?
  time_span_days: number        // Over what time period?
  correlation_score: number     // 0.0-1.0: how strongly correlated with outcome?
  alternative_explanations: string[] // What else could explain this?
  counter_examples: number      // Cases where the pattern was present but outcome was different
}
```

## 9. The Synthesis Engine

### 9.1 Purpose

The Synthesis Engine is the bridge between raw observations and actionable improvements. It runs periodically (not after every task) because patterns only emerge from accumulation.

### 9.2 Synthesis Pipeline

```
PHASE 1: COLLECT
  Query reflection registry for all reflections since last synthesis run.
  Filter: status = validated or clustered, generated_at > last_synthesis_time.

PHASE 2: INDEX
  Build inverted indices:
    by_subject → { atom_id: [reflection_ids], capability: [reflection_ids], ... }
    by_type → { failure: [reflection_ids], gap: [reflection_ids], ... }
    by_relationship → { concerns_atom: { atom_id: [reflection_ids] }, ... }

PHASE 3: CLUSTER
  For each subject with ≥ MIN_CLUSTER_SIZE reflections:
    Group by observation delta similarity (embedding or keyword overlap)
    Assign cluster ID
    Update each reflection's state to 'clustered'

PHASE 4: ANALYZE
  For each cluster:
    Compute aggregate statistics:
      - Frequency (absolute count and rate over time)
      - Trend (increasing, decreasing, stable)
      - Severity distribution
      - Confidence distribution
      - Affected components (union of all relationships)

PHASE 5: FILTER
  Remove noise:
    - Clusters with frequency < MIN_FREQUENCY (configurable, default 3)
    - Clusters where counter_examples > 50% of observations
    - Clusters already addressed by an active improvement candidate
    - Clusters where all observations are from a single session (anomaly, not pattern)

PHASE 6: HYPOTHESIZE
  For each remaining cluster:
    Generate a hypothesis: "If we {change}, then {outcome} will improve by {delta}"
    Estimate confidence in hypothesis (from cluster confidence)
    Estimate impact (tokens saved, failures avoided, time saved)

PHASE 7: PRODUCE
  For hypotheses meeting PROMOTION_THRESHOLD:
    Create Improvement Candidate
    Link candidate to all source reflections
    Update reflections to state 'signaled'
```

### 9.4 Cluster Similarity

Reflections are clustered by semantic similarity of their observations:

```typescript
function clusterSimilarity(a: Reflection, b: Reflection): number {
  // Same subject → base similarity
  let score = 0
  if (a.subject.type === b.subject.type && a.subject.id === b.subject.id) {
    score += 0.5
  }

  // Same reflection type
  if (a.reflection_type === b.reflection_type) {
    score += 0.2
  }

  // Semantic similarity of observation delta
  score += textSimilarity(a.observation.delta, b.observation.delta) * 0.3

  return score
}

// Clusters form when similarity > THRESHOLD (default 0.7)
```

## 20. Long-Term Evolution

### 20.1 The Learning Curve

```
YEAR 1 (10,000 tasks):
  Foundation established.
  Gap discovery rate: high (2-5/day initially, dropping to 0.5/day)
  Knowledge graph grows from ~80 atoms to ~200 atoms
  Compiler parameters mostly hand-tuned
  Most candidates are additive (filling gaps)

YEAR 2 (50,000 tasks):
  Knowledge stabilizing.
  Gap discovery rate: low (0.1/day)
  Compiler parameters self-tuning within safety bounds
  Candidates shift from additive to refining (improving existing atoms)
  First supersessions: old decisions replaced by better ones based on evidence

YEAR 3 (150,000 tasks):
  Deep patterns emerge.
  Cross-project patterns detected (not just within-project)
  Policy evolution based on accumulated security data
  Knowledge graph may split into core (stable) and adaptive (evolving) layers
  Compiler near-optimal for 95% of task types

YEAR 5 (500,000 tasks):
  Institutional knowledge embedded.
  The framework has "experienced" more engineering work than any human on the team.
  New team members get compiled contexts backed by 500,000 reflections.
  The system knows what works because it observed it working.
  The system knows what fails because it observed it failing — and fixed it.
```

### 20.2 Anti-Fragility Principle

The system becomes anti-fragile — it gets stronger with stress:

- **Failures produce reflections** → Reflections identify patterns → Patterns produce candidates → Candidates improve knowledge → Knowledge prevents future failures
- **Gaps produce reflections** → Reflections document the gap → Candidates propose filling it → New atoms fill the gap → Future tasks don't hit the gap
- **Inefficiencies produce reflections** → Reflections quantify the waste → Candidates propose tuning → Compiler parameters improve → Future tasks are more efficient

Every problem the framework encounters makes it better at preventing that problem. This is the fundamental value proposition: **the framework improves through use, not just through maintenance.**

### 20.4 The End State

After years of evolution, the framework:

- **Knows what works** — Not because someone documented it, but because 10,000 reflections confirmed it.
- **Knows what fails** — Not because of intuition, but because 500 failures clustered into 12 well-understood failure patterns.
- **Tells builders exactly what to do** — Not because rules were written broadly, but because the compiler selects atoms that have been proven effective for this exact task type.
- **Tells reviewers what to check** — Not because of a generic checklist, but because reflections identified the specific failure modes for the patterns being used.
- **Gets better every month** — Not because someone worked on it, but because every execution feeds the reflection graph, and the reflection graph feeds improvement candidates, and improvement candidates become promoted knowledge.

The framework doesn't just store engineering knowledge.

It accumulates engineering experience.
